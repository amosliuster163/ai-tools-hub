🧠 OpenClaw多Agent运行机制详解

共享Gateway vs 独立Gateway架构对比 | 作者：老刘 | 2026年3月31日

一、为什么需要多Agent？

在使用OpenClaw的过程中，我发现了一个痛点：

主助手小金负责日常事务协调，但健康助手康康每次都要手动切换很麻烦。

解决方案：配置第二个飞书BOT绑定康康Agent，让康康独立运行，用户直接跟康康聊健康，不用经过小金。

但问题来了：多个Agent是怎么运行的？是并行还是顺序？

二、当前架构：共享Gateway

OpenClaw默认采用共享Gateway架构，所有Agent共用一个Gateway进程。

                        ┌─────────────────────┐
                        │   Gateway (单一进程) │
                        │      PID: 6397      │
                        └─────────────────────┘
                                  │
               ┌──────────────────┼──────────────────┐
               │                  │                  │
        ┌──────▼──────┐    ┌──────▼──────┐    ┌──────▼──────┐
        │  main Agent │    │ kangkang    │    │ 其他Agent   │
        │   (小金)    │    │  (康康)     │    │  (待扩展)   │
        └─────────────┘    └─────────────┘    └─────────────┘

消息路由机制

当你发送消息时，Gateway根据bindings规则路由到对应Agent：

飞书BOT (main) ──消息──► Gateway路由判断 ──► 匹配bindings规则 ──► main Agent

飞书BOT (kangkang) ──消息──► Gateway路由判断 ──► 匹配bindings规则 ──► kangkang Agent

Agent配置示例

Agent Workspace 模型 绑定渠道
main（小金） ~/.openclaw/workspace glm-5 feishu:main
kangkang（康康） ~/.openclaw/workspace-kangkang glm-5 feishu:kangkang

运行特点

特性 说明
进程级别 共享Gateway，单一进程
Session级别 独立Session，对话互不干扰
执行顺序 排队处理，不是真正并行
Memory级别 独立Workspace，记忆互不干扰

三、并发场景分析

如果你同时给两个Agent发消息，会发生什么？

时间T0: 用户 → 小金："帮我查天气"
时间T0: 用户 → 康康："记录体重77kg"

Gateway行为：
1. 接收两条消息（几乎同时）
2. 路由到对应Agent Session
3. 排队处理（小金先处理完 → 康康再处理）
4. 返回结果给各自BOT

实际感受：
- 用户感觉是"同时回复"
- 但内部是顺序执行（时间差很小）
关键点：共享Gateway不是真并行，而是排队处理。但由于处理速度快，用户几乎感觉不到时间差。

四、共享Gateway优劣势

✅ 优势

  • 节省资源（单一进程，内存占用低）
  • 快速响应（排队处理快）
  • 独立记忆（Workspace隔离）
  • 配置简单（一个Gateway管理所有Agent）
  • 统一监控（单一进程便于管理）

❌ 劣势
  • 不是真并行（复杂任务可能排队等待）
  • 共享资源（高负载时可能影响响应）
  • 单点故障（Gateway崩溃影响所有Agent）
  • 资源竞争（多个Agent共享Gateway内存）

五、独立Gateway架构

如果需要真正的并行，可以采用独立Gateway架构：

┌─────────────────┐      ┌─────────────────┐
│ Gateway 1       │      │ Gateway 2       │
│ (独立进程)      │      │ (独立进程)      │
│ PID: 1001       │      │ PID: 1002       │
└─────────────────┘      └─────────────────┘
        │                        │
        ▼                        ▼
┌─────────────────┐      ┌─────────────────┐
│ main Agent      │      │ kangkang Agent  │
│ (小金)          │      │ (康康)          │
│ 独立内存        │      │ 独立内存        │
│ 独立端口        │      │ 独立端口        │
└─────────────────┘      └─────────────────┘

✅ 优势

  • 真并行（两个Agent真正同时执行任务）
  • 资源隔离（一个Agent崩溃不影响其他）
  • 独立端口（可以分别配置外部访问）
  • 负载分散（高负载时不会互相影响）
  • 独立日志（方便排查单个Agent问题）

❌ 劣势

  • 内存翻倍（每个Gateway独立内存约500MB）
  • 配置复杂（需要分别启动、分别配置）
  • 端口冲突（需要分配不同端口）
  • 监控分散（需要分别监控多个Gateway）
  • 重启繁琐（更新时要重启多个Gateway）

六、架构对比总结

架构类型 进程数 内存占用 并行能力 配置复杂度 硬件要求
单Agent单进程 1 低 无 低 ⭐⭐
多Agent共享Gateway 1 中 伪并行 低 ⭐⭐
多Agent独立Gateway N 高 真并行 高 ⭐⭐⭐⭐

七、硬件配置要求

Mac mini硬件评估（实测）

项目 共享Gateway 独立Gateway（2个） Mac mini配置
内存占用 ~500MB ~1GB（500MB × 2） 8GB+ ✅ 充足
CPU占用 单进程 多进程（略高） Apple Silicon ✅ 充足
存储空间 ~1GB ~2GB SSD ✅ 充足
网络端口 单端口（18789） 双端口（18789 + 18790） 家庭网络 ✅ 充足

结论：Mac mini硬件完全支持独立Gateway，硬件不是瓶颈。

重要结论：独立Gateway的瓶颈不是硬件，而是配置复杂度。对于个人用户，共享Gateway架构足够用。

八、适用场景判断

使用场景 是否需要独立Gateway 说明
2-3个Agent，日常使用 ❌ 不需要 共享架构够用，响应快
5+个Agent，高频使用 ✅ 建议独立 避免排队等待
Agent任务复杂（长耗时） ✅ 建议独立 避免阻塞其他Agent
生产环境（高可用要求） ✅ 强烈建议 资源隔离，单点故障不影响全局
个人测试/学习 ❌ 不需要 共享架构足够

九、独立Gateway配置示例

目录结构

~/.openclaw/
├── main/
│   ├── openclaw.json      (main Gateway配置)
│   ├── logs/              (main Gateway日志)
│   └── workspace/
│
└── kangkang/
│   ├── openclaw.json      (kangkang Gateway配置)
│   ├── logs/
│   └── workspace-kangkang/

启动命令

# main Gateway (端口18789)
cd ~/.openclaw/main
openclaw gateway start --config openclaw.json --port 18789

# kangkang Gateway (端口18790)
cd ~/.openclaw/kangkang
openclaw gateway start --config openclaw.json --port 18790

飞书配置变化

BOT Gateway端口 Webhook URL
main BOT 18789 http://服务器IP:18789/webhook
kangkang BOT 18790 http://服务器IP:18790/webhook

十、总结

核心结论

当前建议 保持共享Gateway（2个Agent够用）
切换时机 Agent > 4个 或 出现排队延迟
硬件瓶颈 不是硬件，而是配置复杂度

实践建议

  • ✅ 从共享Gateway开始（配置简单）
  • ✅ 监控响应速度（出现延迟再考虑切换）
  • ✅ 根据实际负载决定（不是预设规划）
  • ✅ 优先优化Agent数量和任务复杂度
